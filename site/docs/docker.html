Docker & CI Run the recorder server in containers and CI pipelines

The recorder server needs Xvfb and ffmpeg, which makes Docker ideal for CI. Your test code runs outside the container and talks to it over HTTP.

Architecture reminder The recorder server (with Xvfb + ffmpeg) runs inside the container. Your tests run wherever you like and connect via an SDK or curl. Only the server container needs system dependencies.

Dockerfile

FROM python:3.12-slim

# System deps for virtual display + recording
RUN apt-get update && apt-get install -y --no-install-recommends \
    xvfb \
    ffmpeg \
    fonts-liberation \
    && rm -rf /var/lib/apt/lists/*

# Install thea-recorder
RUN pip install --no-cache-dir thea-recorder

# Recording output directory
RUN mkdir -p /recordings
VOLUME /recordings

EXPOSE 9123

ENTRYPOINT ["thea", "serve", \
    "--port", "9123", \
    "--output-dir", "/recordings"]dockerfile

Build it:

docker build -t thea-recorder .bash

Docker run

docker run -d \
  --name recorder \
  --shm-size=2g \
  -p 9123:9123 \
  -v $(pwd)/recordings:/recordings \
  thea-recorderbash
Important Always set --shm-size=2g or higher. Xvfb and Chrome use shared memory extensively. Without it, you will get crashes or blank recordings.

Then from your host (or any machine that can reach the container):

# Health check
curl http://localhost:9123/health

# Or use any SDK
thea --server http://localhost:9123 healthbash

Docker Compose

Run the recorder server alongside your test suite:

# docker-compose.yml
services:
  recorder:
    image: thea-recorder
    shm_size: "2g"
    ports:
      - "9123:9123"
    volumes:
      - ./recordings:/recordings
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:9123/health"]
      interval: 5s
      timeout: 3s
      retries: 10

  tests:
    build: .
    depends_on:
      recorder:
        condition: service_healthy
    environment:
      - THEA_URL=http://recorder:9123
    volumes:
      - ./recordings:/recordingsyaml

Run everything:

docker compose up --abort-on-container-exit --exit-code-from testsbash

After the run, recordings are in ./recordings/ on your host.

GitHub Actions

The simplest setup uses a service container to run the recorder alongside your tests. After the run, upload the recordings and HTML report as artifacts so the whole team can review them. 

# .github/workflows/e2e.yml
name: E2E Tests
on: [push, pull_request]

jobs:
  e2e:
    runs-on: ubuntu-latest

    services:
      recorder:
        image: ghcr.io/mechanicalrock/thea-recorder:latest
        ports:
          - 9123:9123
        options: --shm-size=2g

    steps:
      - uses: actions/checkout@v4

      - name: Wait for recorder
        run: |
          for i in $(seq 1 30); do
            curl -sf http://localhost:9123/health && break
            sleep 1
          done

      - name: Run tests
        env:
          THEA_URL: http://localhost:9123
        run: |
          pip install thea-recorder
          pytest tests/

      - name: Download recordings
        if: always()
        run: |
          mkdir -p recordings
          # List all recordings and download each video
          for name in $(curl -s http://localhost:9123/recordings | jq -r '.[].name'); do
            curl -s "http://localhost:9123/recordings/$name/video" -o "recordings/$name.mp4"
          done
          # Download the HTML report
          curl -s http://localhost:9123/report -o recordings/report.html

      - name: Upload recordings & report
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: test-recordings
          path: recordings/yaml
Tip Use if: always() on the download and upload steps so you get recordings even when tests fail. Failed test recordings are the ones you most want to watch.

After the workflow completes, the test-recordings artifact appears on the Actions run summary page. Download the zip — it contains every MP4 video and the interactive HTML report. Open report.html in a browser, click any test step, and the video seeks to that exact moment.

Full GitHub Actions Example

A more complete example showing how to use the Python SDK inside your test code, record multiple scenarios, and publish both videos and the report as downloadable artifacts. 

# .github/workflows/e2e-full.yml
name: E2E with Recordings
on:
  pull_request:
  push:
    branches: [main]

jobs:
  e2e:
    runs-on: ubuntu-latest

    services:
      recorder:
        image: ghcr.io/mechanicalrock/thea-recorder:latest
        ports:
          - 9123:9123
        options: --shm-size=2g --health-cmd "curl -f http://localhost:9123/health" --health-interval 5s --health-retries 10

    steps:
      - uses: actions/checkout@v4

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: "3.12"

      - name: Install dependencies
        run: |
          pip install thea-recorder pytest

      - name: Start display & add panels
        run: |
          # Start the virtual display
          thea --server http://localhost:9123 start-display
          # Add a panel to show what's happening
          thea --server http://localhost:9123 add-panel editor "App Under Test" 80

      - name: Run test suite
        env:
          THEA_URL: http://localhost:9123
        run: |
          pytest tests/ -v --tb=short

      - name: Collect recordings
        if: always()
        run: |
          mkdir -p artifacts/videos
          # Download every recording as MP4
          for name in $(curl -s http://localhost:9123/recordings | jq -r '.[].name'); do
            echo "Downloading $name..."
            curl -s "http://localhost:9123/recordings/$name/video" \
              -o "artifacts/videos/$name.mp4"
          done
          # Download the HTML report
          curl -s http://localhost:9123/report -o artifacts/report.html
          echo "Downloaded $(ls artifacts/videos/*.mp4 | wc -l) recordings"

      - name: Upload videos
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: e2e-videos
          path: artifacts/videos/
          retention-days: 14

      - name: Upload report
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: e2e-report
          path: artifacts/report.html
          retention-days: 30yaml
How it works
  1. The recorder runs as a service container — GitHub provisions it alongside your job.
  2. Your tests talk to it on http://localhost:9123 using any SDK or curl.
  3. Each test starts/stops a recording. The recorder keeps videos in memory.
  4. After tests finish, you download videos via HTTP API — /recordings/{name}/video.
  5. The HTML report is served at /report and embeds clickable timestamps per step.
  6. upload-artifact makes everything available on the Actions summary page.

Using the SDK in your tests

Here's what a test file might look like with the Python SDK:

import os
from thea import RecorderClient

client = RecorderClient(os.environ["THEA_URL"])

def test_invoice_submission():
    client.start_recording("invoice-submission")
    client.update_panel("editor", "Submitting invoice INV-042...")

    # ... your actual test code here ...

    client.update_panel("editor", "Invoice submitted successfully")
    result = client.stop_recording()
    assert result["path"]

def test_tax_calculation():
    client.start_recording("tax-calculation")
    client.update_panel("editor", "Running tax calculation...")

    # ... your actual test code here ...

    result = client.stop_recording()
    assert result["path"]python

Each test produces its own named recording. After the suite finishes, the collect step downloads all videos and the report in one pass.

Viewing artifacts

After the workflow runs, go to the Actions tab in your repo, click the run, and scroll to Artifacts. You'll see:

  • e2e-videos — a zip of all MP4 recordings, one per test scenario
  • e2e-report — the interactive HTML report you can open locally

Share the report link with QA, attach it to PRs, or use it to investigate failures. The video evidence makes it easy to communicate exactly what happened.

GitLab CI

# .gitlab-ci.yml
e2e:
  image: python:3.12
  services:
    - name: thea-recorder
      alias: recorder
  variables:
    THEA_URL: http://recorder:9123
  before_script:
    - pip install thea-recorder
    - |
      for i in $(seq 1 30); do
        curl -sf $THEA_URL/health && break
        sleep 1
      done
  script:
    - pytest tests/e2e/
  artifacts:
    when: always
    paths:
      - recordings/yaml

Tips

  • Always set --shm-size=2g (or higher) in Docker. Chrome and Xvfb need shared memory. Without it you will get crashes.
  • Mount a volume for the recordings directory so you can access videos after the container exits.
  • Use the healthcheck in Compose. The SDK clients now auto-wait for the server on their first API call, so explicit wait_until_ready() calls are no longer needed.
  • Graceful shutdown works. The server handles SIGTERM, so docker stop will not lose a recording in progress.
  • Upload artifacts on failure. Always upload recordings even when tests fail — those are the videos you most want to watch.
  • Font rendering: Install fonts-liberation or similar in your image for readable text in panel overlays.
