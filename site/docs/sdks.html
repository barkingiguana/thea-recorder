SDKs Thin HTTP client libraries for every language

Each SDK wraps the REST API with idiomatic method calls, error handling, and convenience helpers. Every SDK follows the same pattern: connect, record, done.

Behave integration

# features/environment.py
from thea import RecorderClient

client = RecorderClient("http://localhost:9123")

def before_all(context):
    client.start_display()
    client.add_panel("status", title="Status", width=120)
    client.add_panel("scenario", title="Scenario")

def before_scenario(context, scenario):
    client.update_panel("status", "Running")
    client.start_recording(scenario.name)

def after_scenario(context, scenario):
    result = client.stop_recording()
    print(f"Video: {result['path']}")

def after_all(context):
    client.cleanup()python

godog integration

func InitializeTestSuite(ctx *godog.TestSuiteContext) {
    ctx.BeforeSuite(func() {
        client = thea.NewClient("http://localhost:9123")
        c := context.Background()
        client.StartDisplay(c)
        client.AddPanel(c, "status", "Status", intPtr(120))
    })

    ctx.AfterSuite(func() {
        client.Cleanup(context.Background())
    })
}

func InitializeScenario(ctx *godog.ScenarioContext) {
    ctx.Before(func(ctx context.Context, sc *godog.Scenario) (context.Context, error) {
        client.StartRecording(ctx, sc.Name)
        return ctx, nil
    })

    ctx.After(func(ctx context.Context, sc *godog.Scenario, err error) (context.Context, error) {
        result, _ := client.StopRecording(ctx)
        fmt.Printf("Video: %s (%.1fs)\n", result.Path, result.Elapsed)
        return ctx, nil
    })
}go

Bash / curl

No SDK needed. Use the thea CLI or plain curl calls.

CLI quick start

export THEA_URL=http://localhost:9123

thea start-display
thea add-panel --name status --title Status --width 120
thea start-recording --name login_test
thea update-panel --name status --text "Running"

# ... run your tests ...

thea update-panel --name status --text "PASSED"
thea stop-recording
thea cleanupbash

curl quick start

# Start display
curl -X POST http://localhost:9123/display/start

# Start recording
curl -X POST http://localhost:9123/recording/start \
  -H "Content-Type: application/json" \
  -d '{"name": "login_test"}'

# ... run your tests ...

# Stop recording
curl -X POST http://localhost:9123/recording/stop

# Cleanup
curl -X POST http://localhost:9123/cleanupbash

Common API Surface

Every SDK provides these methods (names may be camelCase or snake_case per language convention):

MethodDescription
start_display()Launch Xvfb
stop_display()Stop Xvfb
add_panel(name, title, width)Add overlay panel
update_panel(name, text, focus_line)Update panel content
remove_panel(name)Remove panel
list_panels()List all panels
start_recording(name)Begin recording
stop_recording()Stop recording, get path + elapsed
recording_elapsed()Get elapsed seconds
recording_status()Get recording state
list_recordings()List available MP4 files
download_recording(name, path)Download MP4 to local file
recording_info(name)Get file metadata
health()Server health check
cleanup()Full teardown

Convenience Helpers

Every SDK provides scoped helpers that handle start/stop automatically:

HelperDescription
recording(name)Start recording, run code, stop recording (even on error)
with_panel(name, ...)Add panel, run code, remove panel
wait_until_ready(timeout)Poll /health until server responds. Called automatically on first API call.
Tip Always use recording() instead of manual start/stop. It handles cleanup on test failure so you never lose a recording or leave the server in a bad state.
